# Harness DDD/TDD Upgrade Research

## Executive Summary

Harness already has useful gates, but DDD and TDD are not yet connected as one traceable workflow. The upgrade should make each change flow through:

`Domain Contract -> Implementation Boundary -> Red Test -> Green Implementation -> QA Evidence`.

The strongest evidence supports four changes:

- Add a DDD/TDD gate reference document for all Harness steps.
- Strengthen Step 2 so every domain artifact records public contracts, invariants, context relationships, and executable examples.
- Strengthen Step 3 so implementation plans include a traceability matrix and test design matrix.
- Strengthen Step 4, red-test templates, and test guides so tests come from the domain contract before code, including contract-violating cases for LLM-generated code.

## Scope And Assumptions

Scope: upgrade the local Harness skill documentation and templates so future `/harness` runs apply DDD and TDD more rigorously.

Assumptions:

- Do not add a new runtime dependency such as Pact or Context Mapper.
- Use existing Harness artifacts and gates.
- Keep the change procedural and small enough for current Harness workflows.

## Methodology

Selected tier: Standard. I ran 13 research passes across DDD strategy, tactical DDD, event storming, bounded-context canvases, TDD, test pyramids, test sizing, contract testing, flaky-test evidence, and LLM contract adherence.

## Key Findings

DDD must start with bounded contexts and shared language. Fowler describes bounded context as central to strategic DDD, dividing larger models into explicit contexts with clear relationships; he also frames ubiquitous language as a rigorous shared language between developers and users ([Bounded Context](https://martinfowler.com/bliki/BoundedContext.html), [Ubiquitous Language](https://martinfowler.com/bliki/UbiquitousLanguage.html)).

Domain analysis should precede technology choice. Microsoft’s domain-analysis guidance says to design around business capabilities, not horizontal layers, and to use bounded contexts, entities, aggregates, and services to keep systems cohesive and loosely coupled ([Azure Domain Analysis](https://learn.microsoft.com/en-us/azure/architecture/microservices/model/domain-analysis), [Tactical DDD](https://learn.microsoft.com/uk-ua/azure/architecture/microservices/model/tactical-domain-driven-design)).

The bounded-context artifact should expose interfaces and uncertainties. DDD Crew’s Bounded Context Canvas emphasizes responsibilities, public interface, dependencies, verification metrics, and open questions; it also notes that a public interface is the contract with the rest of the system ([Bounded Context Canvas](https://github.com/ddd-crew/bounded-context-canvas)).

TDD needs a test list before implementation. Fowler’s TDD description adds an initial test-case list before the red-green-refactor loop, and says test-first thinking forces interface design first ([Test Driven Development](https://martinfowler.com/bliki/TestDrivenDevelopment.html)).

Test strategy should mix granularity. Fowler’s practical test pyramid advises many small tests, some coarse-grained tests, and very few high-level E2E tests; he also warns that E2E tests are flaky and costly ([Practical Test Pyramid](https://martinfowler.com/articles/practical-test-pyramid.html)).

Test size constraints are useful because they are enforceable. Google’s test-size model distinguishes small, medium, and large tests by network, database, filesystem, threading, sleeps, and time limits, and says tests should run in any order with high isolation ([Google Test Sizes](https://testing.googleblog.com/2010/12/test-sizes.html)). Google also reports flakiness rising sharply with test size ([Where Do Our Flaky Tests Come From?](https://testing.googleblog.com/2017/04/where-do-our-flaky-tests-come-from.html)).

Contracts need executable examples. Pact documentation describes consumer-driven contract tests as code-first examples generated by consumer tests and verified by providers ([Pact Introduction](https://docs.pact.io/)). Harness should use the pattern without requiring Pact.

LLM coding needs contract-violation tests. ContractEval reports that code models can score high on normal functional correctness yet fail contract satisfaction, making explicit contract enforcement a separate quality axis ([ContractEval](https://arxiv.org/abs/2510.12047)). Related LLM test-generation research also shows that generated tests still miss requirement alignment and coverage gaps ([Text-to-Testcase Generation](https://arxiv.org/abs/2402.11910), [Examination of Code Generated by LLMs](https://arxiv.org/abs/2408.16601)).

## Recommendations

1. Add a shared `ddd-tdd-gates.md` reference and link it from Step 2 through Step 6.
2. Require Step 2 `Domain Contract` to include bounded context, ubiquitous language, public contracts, invariants, commands/events, context relationships, and contract examples.
3. Require Step 3 to map each domain contract item to files, implementation steps, red tests, green commands, and QA evidence.
4. Require Step 4 red artifacts to be created from the contract and include at least one contract-violating case where there is an API, IPC, persistence, permission, validation, or state boundary.
5. Update test-guide format so QA validates domain examples and contract violation cases, not only happy paths.

## Risks, Uncertainties, And Contradictions

- DDD can become ceremony if Harness forces aggregates and repositories for simple tasks. Mitigation: require contract and boundary thinking, but do not require tactical DDD patterns unless domain complexity justifies them.
- Pact and Context Mapper are useful but would be too heavy as mandatory dependencies. Mitigation: use their principles as artifact structure only.
- LLM contract research is emerging and some sources are preprints. Mitigation: use it to justify guardrails, not to claim exact model behavior for every project.
