# Harness 금지 사항

## 범위

- 사용자가 요청하지 않은 기능을 추가하지 않는다.
- 인접 코드 정리나 대규모 리팩터링을 Harness 단계에 끼워 넣지 않는다.
- 산출물 경로를 현재 worktree 기준으로 추측하지 않는다. 항상 Git common dir 기준 메인 repo 루트를 사용한다.
- worktree 안에 별도 `.harness`를 만들지 않는다.
- 외부 문서의 `Step 12`, `Step 22`, `Task 3` 같은 번호를 Harness Step으로 부르지 않는다. Harness 번호는 Step 1-8과 Complete만 사용한다.

## 질문 정책

- `/harness`는 noask 모드다. 질문 대신 합리적 가정과 보류 사유를 progress 파일에 남긴다.
- `/harness-ask`에서만 결정 질문을 한다.
- noask 모드에서 테스트 미실행, 리뷰 미완료, 환경 차단을 PASS로 바꾸지 않는다.

## 검증

- 실행하지 않은 명령을 실행했다고 쓰지 않는다.
- 화면을 보지 않은 UI 검증을 완료로 쓰지 않는다.
- 검증 없는 자기 확신을 `PASS`로 쓰지 않는다.
- `PASS`는 명령 출력, 로그, 스크린샷, 브라우저 검증, 테스트 리포트처럼 재확인 가능한 증거가 있어야 한다.
- 핵심 흐름을 실행하지 못했으면 `PASS`가 아니라 `UNKNOWN` 또는 `BLOCKED`다.
- 실행한 검증과 실행하지 못한 검증을 분리해 적는다.
- QA와 고객 테스트 도우미는 보고서와 스크린샷 외 구현 파일을 수정하지 않는다.

## Git

- 명시 요청 없이 destructive git 명령을 실행하지 않는다.
- commit/push는 사용자 요청 또는 워크플로 단계에서 명확히 요구된 경우에만 한다.
