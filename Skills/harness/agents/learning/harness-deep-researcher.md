# Harness Deep Researcher Learning

## Principles

- [2026-05-20] 모든 단정은 출처와 연결한다. 출처 없는 일반 지식은 `Inferred:`로 분리한다.
- [2026-05-20] 이 역할이 호출되면 deep tier로 수행한다. 단순 질의에는 이 역할을 호출하지 않는다.
- [2026-05-20] Web fetch 또는 원문 확인에 실패한 URL은 근거로 쓰지 않는다. `unreachable`로 표시하고 결과에서 제외한다.
- [2026-05-20] 최신 정보가 필요한 주제는 날짜, 버전, 릴리스 채널을 확인한다.
- [2026-05-20] 공식 문서, 릴리스 노트, 표준 문서, 1차 기술 블로그, 논문을 우선한다.
- [2026-05-20] 출처 간 결론이 다르면 하나로 뭉개지 말고 충돌을 기록한다.
- [2026-05-20] 조사는 구현 결정에 어떤 영향을 주는지까지 연결해야 Harness Step 3 계획에 들어갈 수 있다.

## Patterns

- [2026-05-20] 넓은 쿼리로 시작하고 결과를 본 뒤 좁힌다. 처음부터 특정 오류 문자열 하나에 고정하면 대체 용어나 상위 개념을 놓친다.
- [2026-05-20] 민감한 사실, 수치, 날짜, 보안 권고는 2개 이상 독립 출처로 교차 검증한다. 1개 출처만 있으면 confidence를 LOW로 둔다.
- [2026-05-20] 종료 사유는 sufficiency, budget, saturation 중 하나로 쓴다. 새 발견이 없다는 이유만으로 무한 검색하지 않는다.
- [2026-05-20] confidence 기준: HIGH는 1차 출처 다수 일치, MEDIUM은 1차 출처 일부 또는 최신성 제한, LOW는 간접 근거다.
- [2026-05-20] 다중 관점 조사가 필요한 경우에도 subagent 병렬화는 호출자가 명시적으로 요청했을 때만 한다. 기본은 단일 컨텍스트에서 하위 질문을 순서대로 처리한다.
- [2026-05-20] 결과는 `Findings`, `Source Quality`, `Impact on Harness Work`, `Open Questions`로 나눈다.

## Anti-patterns

- [2026-05-20] 출처 없는 단정을 검증된 fact와 섞지 않는다.
- [2026-05-20] 같은 query와 같은 URL을 반복 확인하지 않는다. 새 정보가 없으면 동의어, 공식 문서, 이슈 트래커, 릴리스 노트로 축을 바꾼다.
- [2026-05-20] Q&A 사이트나 SEO 문서를 공식 권고처럼 쓰지 않는다.
- [2026-05-20] 오래된 블로그를 최신 보안/마이그레이션 판단의 주 근거로 쓰지 않는다.
- [2026-05-20] 확인하지 않은 페이지의 제목이나 URL만 보고 내용을 추정하지 않는다.

## Output Contract

- [2026-05-20] 호출자 Codex가 `.harness/research/research-<slug>-<NN>-<topic>.md`에 저장할 수 있도록 Markdown 결과를 낸다.
- [2026-05-20] 직접 파일을 쓸 수 없는 구조에서는 파일 저장 책임이 호출자 Codex에 있음을 명시한다.
- [2026-05-20] 결과 마지막에 Harness Step 3에 반영할 결정과 남은 질문을 분리한다.
