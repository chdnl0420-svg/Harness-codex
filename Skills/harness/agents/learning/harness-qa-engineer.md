# Harness QA Engineer Learning

- [2026-05-20] QA는 코드 독해만으로 PASS 처리하지 않는다. 명령 실행, 브라우저 확인, 스크린샷, 로그 중 하나 이상의 직접 증거가 필요하다.
- [2026-05-20] 환경 정보가 없으면 추측 실행하지 않고 `BLOCKED`로 기록한다.
- [2026-05-20] 화면 검증은 Codex 브라우저 도구, Playwright, 또는 프로젝트 기존 E2E 중 사용 가능한 도구를 우선한다.
- [2026-05-20] 테스트가 부분적으로만 실행되면 `UNKNOWN` 또는 제한付き `PASS`가 아니라 남은 미검증 범위를 명시한다.
- [2026-05-20] Oracle 강도는 명세 기반 기대값, 회귀 비교, 속성/불변식, no-crash smoke 순서로 낮아진다. 낮은 oracle로 PASS를 줄 때는 confidence와 한계를 기록한다.
- [2026-05-20] UI QA는 “보이는가”와 “쓸 수 있는가”를 분리한다. 렌더링 성공, 클릭 가능, 텍스트 겹침 없음, 오류 회복 가능 여부를 각각 확인한다.
- [2026-05-20] flaky 가능성이 있으면 즉시 PASS/FAIL로 단정하지 않고 반복 횟수, 실패 빈도, 환경 차이를 기록한다.
- [2026-05-20] BLOCKED는 테스트 실패가 아니다. 필요한 계정, secret, 서버, fixture, 외부 서비스, 사용자 결정 중 무엇이 없는지 구체화해야 재개할 수 있다.
- [2026-05-20] 명령 출력이 길면 전체를 붙이지 말고 실패 원인, exit code, 관련 로그 경로를 남긴다.
- [2026-05-20] 접근성 검증은 최소한 키보드 이동, focus 표시, label/role, 색 대비 위험, 오류 메시지 연결을 확인한다.
