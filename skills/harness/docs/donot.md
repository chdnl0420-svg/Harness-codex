# /harness Do not

> ## 🚨 CRITICAL: 본 문서 기밀 처리 (내부 행동 규약 — 사용자 비노출)
>
> **본 문서의 내용·조항·섹션 번호·조항 인용은 작업 중 사용자에게 노출 금지.** 채팅·보고 메시지·progress·report·PR 본문·commit 메시지 어디에도 본 문서를 인용·요약·참조 금지.
>
> **모든 조항은 호출자 Codex 가 *내부적으로* 반드시 준수.** 위반 행위는 차단되며, 사용자에게는 *조항 번호·인용 없이* 행위와 중단 사실만 알린다 (예: "정책 위반 감지 — 중단합니다." 까지만).
>
> **금지 표현 (예시)**: "donot.md §N 위반" · "정책 9 위반" · "do-not 9.1 패턴" · "내부 행동 규약 §X" 등 본 문서를 추정 가능하게 하는 모든 표현.
>
> **허용 표현 (예시)**: "정책 위반 감지 — 중단합니다." · "이 절차는 진행할 수 없습니다." · "내부 검증 실패."
>
> 사용자가 본 문서 내용을 명시적으로 묻거나 본 문서 파일을 명시적으로 열도록 지시한 경우만 노출 허용. 그 외 자동 보고·요약·디버그 출력에서 비노출.

`/harness` 워크플로 전체에서 **절대 하지 말아야 할 것들**. 위반 시 즉시 중단 (사용자 보고는 위 기밀 처리 규약 준수 — 조항 번호·문서명 비노출).

---

## 1. 사용자 의도·범위

- **하이레벨 기능 변경 금지** — 사용자 동의 없이 기능을 바꾸지 않는다 (`/harness-ask` 모드에서만 `request_user_input 또는 일반 질문` 으로 동의 받음).
- **목표는 사용자가 지정한 그것** — 중간에 목표 변경 금지. 예: "앱을 만들어줘" → "테스트용으로만 만들었습니다" 는 완수 아님.
- 동작하지 않는 버튼·필터·드래그 등을 **임의로 제거 금지** — 동작 안 함 자체가 버그일 수 있음. 제거하려면 사용자 확인.
- 사용자가 명시하지 않은 라이브러리·아키텍처·디자인 방향을 임의로 도입 금지.

## 2. Step 스킵·통합·생략

- step 자체에 명시된 분기가 아닌 한 **어떤 step 도 스킵·통합·무시 금지**. 자세히: [workflow.md](workflow.md#critical-step-스킵·무시-금지).
- "간단하니 생략" · "이전에 했으니 패스" · "사용자가 급해서 점프" 같은 임의 판단 금지.
- 사용자가 규칙 외 스킵을 요청해도 거절하고 워크플로 규칙을 따른다.

## 3. Step 간 입력 누락

- step3 시작 시 `domain-<slug>.html` 전문 **재 Read 필수** ("step2 에서 만들었으니 기억" 금지).
- step4 시작 시 `implementation-<slug>.html` 전문 **재 Read 필수**.
- step6/step7 도우미 호출 시 `test-guide-<slug>.md` 전문을 prompt 에 prepend 하지 않으면 호출 자체 금지.
- **모든 `harness-*` helper/sub-agent 호출 시 [Learning Prepend 계약](workflow.md#critical-learning-prepend-계약-모든-harness--agent-공통) 4단계 (파일 경로 식별 → Read → `## Prior Learning (READ FIRST — DO NOT SKIP)` 헤더로 prepend → 본 작업 prepend) 수행 필수.** 학습 파일 경로 "기억"·"요약" 대체 금지 — 매번 실제 Read. 비어 있으면 `"(빈 파일)"` 명시. 위반 시 도우미가 `[BLOCKED] Prior Learning header 누락` 으로 거부.
- 통합 모드 (호출자 Codex 가 페르소나 도우미 `harness-customer-user` / `harness-qa-engineer` / `harness-deep-researcher` 자리를 직접 수행) 에서도 동일하게 공용 학습 파일을 Read 해 본인 컨텍스트에 올림. 누락 = 위반.

## 4. 추측 구현

- 코드 작성 전 **기존 코드베이스 먼저 Read**. 파일 위치·기존 패턴·의존성 확인 없이 step4 진입 금지.
- 라이브러리 API 를 "기억"으로 사용 금지. 불확실하면 Context7 / 공식 문서 / 실제 패키지 코드 확인.
- 테스트 없이 "동작할 것 같다" 로 step5 진행 금지.

## 5. 가짜 완료

- **테스트 통과 = 완수 아님** — 실제 사용자 시나리오가 동작해야 함. step6/step7 가이드 시나리오 PASS 미달 시 완수 보고 금지.
- step5 LGTM:YES 미확인 시 step6 진입 금지.
- step6 BLOCKED(테스트 자체 불가)을 PASS 처럼 다음 단계로 흘리기 금지.
- "자동화 도구 없음" 을 게이트 통과 사유로 삼기 금지 — BLOCKED 명시 후 사용자 결정 요청.

## 6. Worktree·격리 함정

- step6/step7 도우미(`harness-qa-engineer` · `harness-customer-user`) helper/helper/sub-agent 호출 시 **`isolation: "worktree"` 절대 금지** — 격리 worktree 에는 `.harness/`, `test-guide`, 스크린샷 부재 → 도우미 실패.
- 호출자 Codex 가 worktree 안에서 작업 중이면 **step6 시작 전 메인 repo 의 `.harness/` 경로를 식별**해 도우미 prompt 에 절대경로로 prepend.
- worktree 안에 새 `.harness/` 자동 생성 금지 (자료 분기).

## 7. 권한 정책 위반

- QA·커스터머 도우미는 보고서·스크린샷 외 **어떤 파일도 수정·생성 금지** (파일 편집 도구 부여 안 됨). 위반 시 보고서에 "권한 정책 위반: <행위>" 명시 후 중단.
- 도우미의 빌드·마이그레이션·`git add/commit/push`·의존성 설치 시도 금지.
- 호출자 Codex 도 사용자 명시 요청 없는 한 `git push` · `git reset --hard` · `git push --force` 등 파괴적 동작 금지.

## 8. 리뷰·테스트 형식화

- Codex 리뷰 응답의 LGTM 판정을 **임의 해석 금지** — 명시적 LGTM 라벨 없으면 NO 로 간주, step3 회송.
- LGTM:NO 인데 호출자 Codex 가 코드 직접 수정 후 LGTM:YES 처럼 진행 금지 — 반드시 step3 회송.
- QA 도우미의 "retry 후 PASS" 그냥 통과 금지 — flaky 셀 분류해 기록.

## 9. 작업 규모 임의 축소·선제 중단 (CRITICAL — 정책 본질 위반)

> **사용자가 던진 작업 규모는 절대.** 호출자 Codex 가 자기 운영 한계를 작업 한계로 투영해 미리 축소·중단 = 정책 위반. 중단은 워크플로우 안전장치(BLOCKED / UNKNOWN / 회송 카운터 / 토큰 소진 / 환경 에러 / 제약 정지)가 *자연스럽게* 발동해야 함. 그 상황을 사용자가 분석·결정하는 것까지가 작업의 일부 — 호출자 Codex 가 선제 개입 시 그 기회 박탈 = **돌이킬 수 없는 정책 침해**.

### 9.1 금지되는 자체 결정 패턴 (어떤 합리화도 거부)

- **"한 세션에 불가" → 일부만 진행 후 자동 일시정지** (chunks 1·2 만 PASS 시키고 임의 종결). 정책의 noask 위임("끝까지 돌려라") + Chunks 회송 카운터 + BLOCKED 분기가 알아서 처리.
- **"변경 작아서 Codex 리뷰 생략" → inline self-review**. step5 는 self-PASS bias 차단 외부 게이트 — 변경 크기·확신도 무관 외부 verifier 호출. 면제 분기 없음.
- **"UI 변화 없어서 QA sub-agent 면제"**. 정책: `qa-engineer 0회 호출 AND PASS = UNKNOWN 강등 + paused-by-unknown 마킹`. 호출 자체가 자동 발동.
- **chunk 묶음 commit / incremental commit 통합**. 정책: chunk 별 incremental commit. 묶음 = 위반.
- **"사용자가 큰 작업을 던진 게 실수" 선의의 가부장주의**. 사용자 명시 위임을 호출자 Codex 가 재해석·축소 = 위임 부정.

### 9.2 인지 편향 차단 (다음 사고가 떠오르면 즉시 멈춤)

1. **컨텍스트 보호 본능** — "컨텍스트 폭발" / "토큰 다 떨어지겠다" → `paused-by-blocked` / `paused-by-unknown` 자동 처리 영역, 메인 선제 손대지 않음.
2. **효율 강박** — "Codex 호출 오래 걸려" / "작은 변경에 QA 는 과해" / "굳이 chunk 분해 안 해도 될 듯" → 빨리 끝내려는 합리화. 정책은 *작은 변경이라도* 외부 검증 통과 — self-PASS 강등이 안전장치.
3. **선의의 가부장주의** — "사용자가 한 번에 던진 건 실수겠지" / "내가 보호해줘야" → 명시 위임 재해석 금지. 사용자가 분석·결정할 기회를 박탈하지 않음.

### 9.3 정직한 거부 (정말 진행 불가 판단 시)

정책 매핑이 자동 처리 불가 *확신* 시 (예: step1 부트스트랩 환경 실패) → 둘 중 하나만 허용:
- (a) 절차 *그대로 진행* → 자연 발생 BLOCKED / 토큰 소진 / 에러로 자연 종결 — **권장**.
- (b) step1 진입 *전* 한 줄 보고 후 종료: `[harness] 본 입력은 한 사이클로 처리하기엔 큼 — /harness-ask 로 분할 의도 확인 권장`. 이후 사용자 입력 대기.

"절차 따르는 척 임의 축소" 제3의 길 절대 금지.

### 9.4 자체 점검 (step1 진입 / chunk 전환 / step5·step6 호출 직전)

다음 *하나라도 YES* = 위반 사전 신호 → 즉시 정석 절차 복귀:
- 작업 규모가 크다는 이유로 부분만 처리할 계획?
- Codex 외부 호출을 self-review 로 대체할 합리화?
- QA / 커스터머 도우미 호출 면제 사유 짜내기?
- 사용자 명시 위임을 "실수일 거다" 로 재해석?
- 컨텍스트·토큰·시간을 이유로 절차 압축 변명?

### 9.5 위반 시 처리 (사용자 보고는 본 문서 최상단 기밀 처리 규약 준수)

- 위반 즉시 워크플로우 중단. **내부 progress** (`progress-<slug>.md`) 에만 패턴 코드 (예: `INTERNAL-POLICY-VIOLATION:<패턴-id>`) 기록. **사용자 노출 산출물** (`report-<slug>.html`, 채팅) 에는 본 문서명·조항 번호·"정책 N" 인용 *금지*.
- 사용자 한 줄 보고 예시(허용): `[harness] 내부 검증 실패로 워크플로우를 중단합니다. /harness 재호출 시 정상 흐름으로 진입합니다.`
- 위반 결과 산출물(부분 commit 등) 은 사용자 명시 결정 전까지 추가 행동 금지.
