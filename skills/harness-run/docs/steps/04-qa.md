# step 4 - QA 서브에이전트

`harness-engineering-qa` 서브에이전트가 빌드, 테스트, 커버리지, 정적 분석을 검증한다.

## 호출 전 준비

1. `~/.codex/skills/harness-run/learning/harness-engineering-qa.md` 를 읽어 prompt 맨 앞에 붙인다 (v2.1 — Codex 최상위 agents 산하 single source of truth).
2. 프로젝트 루트, 실행 명령, 산출물 폴더 절대경로를 전달한다.
3. QA 결과를 `.harness/04-qa/qa.md` 에 쓰도록 요구한다.

## 검증 범위

- 빌드 명령 실행
- 전체 테스트 실행
- 커버리지 80% 이상 확인. **coverage 명령이 프로젝트에 설정되지 않은 경우 (도구 부재) → `Verdict: BLOCKED` + `BLOCKED_REASON: EVIDENCE_GATE_FAIL` (서브카테고리 `coverage_tool_missing`)** — 임의로 PASS 작성 금지. enum 은 step 4 본문·QA agent·BLOCKED reason 표 모두에서 동일.
- 린터, 타입체커, 정적 분석 실행
- Mock 금지 위반 탐색
- production credential 또는 production endpoint 호출 위험 재점검
- UI 프로젝트면 View/Logic 분리 위반 탐색

## 판정

QA 보고서는 반드시 다음 중 하나를 포함한다.

- `Verdict: PASS`
- `Verdict: FAIL`
- `Verdict: BLOCKED`

`BLOCKED` 의 경우 다음 2 필드 모두 의무:

- `BLOCKED_REASON` (top-level enum): `DEPENDENCY_MISSING | EVIDENCE_GATE_FAIL | PERMISSION_DENIED | GUIDE_MISSING | ENV_UNREACHABLE | CONTRACT_MISSING | TDD_MISSING | OTHER`
- `BLOCKED_SUBCATEGORY` (자유 텍스트, 가능하면 표준 값 사용):
  - `EVIDENCE_GATE_FAIL` 의 표준 subcategory: `coverage_tool_missing` / `lint_config_missing` / `typecheck_config_missing` / `test_runner_missing` / `mock_detected` / `prod_endpoint_detected`
  - `DEPENDENCY_MISSING` 의 표준 subcategory: `package_install_failed` / `runtime_missing` / `tooling_version_mismatch`
  - 그 외 enum 은 자유 텍스트

QA agent · audit findings template 모두 동일 enum/subcategory 사용.

## 회송

- PASS: step 5 로 진행
- FAIL: 실패 원인을 step 3 입력으로 붙이고 구현 루프로 되돌린다
- BLOCKED: 로그에 사유와 재시도 조건을 기록하고 사용자에게 보고한다

호출자 Codex 는 QA 판정을 직접 작성하지 않는다. 서브에이전트가 파일을 못 쓰면 응답 본문을 그대로 저장한다.

## 충돌 해소 — refactor 회차의 coverage waiver (`run-modes.md` 와의 우선순위)

본 step 4 §검증 범위의 "coverage 명령이 프로젝트에 설정되지 않은 경우 → `Verdict: BLOCKED`" 조항은 `run-modes.md` 의 "refactor 회차 coverage waiver 허용" 과 *충돌하지 않음*. 충돌 해소 우선순위 §6 #3 (회차 유형별 강제 항목) > #5 (step 별 산출물) 적용:

| 상황 | 결과 |
|---|---|
| coverage 도구 *부재* (`package.json` 에 coverage script 없음 등) + `04-qa/coverage-waiver.md` **사전 작성** | **Verdict: PASS_WITH_WAIVERS** (BLOCKED 아님) |
| coverage 도구 *존재* + 측정 결과 80% 미달 + waiver **사전 작성** | **Verdict: PASS_WITH_WAIVERS** (refactor 회차만 허용) |
| coverage 도구 부재 + waiver **미사전 작성** | **Verdict: BLOCKED** (post-hoc waiver 불가) |
| run-mode = `new-domain` | waiver path 금지 — 무조건 BLOCKED (coverage 80% 미달 시) |

waiver.md 의 *사전 작성* 정의: step 4 진입 시점 또는 step 4 종료 ≤ step 5 진입 사이 작성 (post-hoc 분 차이는 audit 가 허용 가능, 그러나 *step 7 audit 이후 작성은 post-hoc 불가*).

`04-qa/coverage-waiver.md` 5 필드: 생략 항목 (coverage 80%) / 사유 (도구 부재 또는 refactor 회차) / 대체 검증 (build EXIT=0, typecheck EXIT=0, smoke 매핑 등) / audit 허용 조건 / non-waivable invariant 충돌 확인.

cycle 4 (`20260525T140541Z-refactor`) 의 step 4 이견 (1차 CONDITIONAL vs 2차 PARTIAL/P1) 가 본 §로 해소된 선례.
