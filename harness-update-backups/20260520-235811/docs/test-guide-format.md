# Test Guide 포맷

`.harness/test-guide-<slug>.md` 파일의 표준 양식.

step6 의 1단계에서 Codex가 이 양식대로 작성/갱신한다.
step6 (QA) / step7 (Customer) 두 도우미가 이 문서를 **유일한 진실 원천** 으로 삼는다.

---

## 양식

```markdown
# Test Guide — <slug>

## 1줄 요약
이 제품/기능이 무엇을 하는지 한 문장.

## 페르소나
- QA: <어떤 사용자를 대신해 테스트하는지>
- Customer: 제품 처음 본 일반인 (개발/도메인 지식 없음)

## 환경
- URL / 진입 경로: ...
- 로그인 / 시드 데이터: ...
- 사전 조건: ...

## 하이레벨 기능 목록
테스트해야 할 항목을 사용자 가치 단위로 나열. 코드 함수 아님.

- F1: <기능 한 줄>
- F2: ...
- F3: ...

## 기능별 정상 흐름
### F1: <이름>
- 입력/조작: ...
- 기대 결과: ...
- 측정 가능한 성공 기준 (예: 메시지 노출, 상태 변경, 응답 시간 등)

### F2: ...

## 경계 / 오용 케이스
- F1 에 빈 입력 / 최대 길이 / 특수문자 ...
- F2 에 잘못된 권한 / 중복 / 동시성 ...

## 회귀 위험 영역
이번 변경이 기존 기능 중 어디를 깨뜨릴 수 있나.
- 기존 기능 X — 어떤 시나리오 다시 돌려야 하나
- 기존 기능 Y — ...

## 비목표 / 테스트 범위 밖
PRD 비목표·MVP 제외 사항. 도우미가 이걸 검증하려고 시간 쓰지 않게.

## 변경 이력
- v1 (YYYY-MM-DD HH:MM) — 최초 작성
- v2 (...) — <어떤 변경>
```

---

## 작성 재료

있는 것만 합침. 없는 항목은 "없음" 으로 명시.

- `docs/PRD.md` — 제품 명세
- `docs/UI_GUIDE.md` — UI/UX 가이드
- `domain-<slug>.md` — 이번 작업의 도메인 설계
- `implementation-<slug>.md` — 이번 작업의 구현 계획
- `review-<slug>.md` — 최신 리뷰 결론
- 사용자가 대화에서 추가한 요구사항

---

## 갱신 규칙

- 사양·구현·리뷰 결과가 바뀌면 **해당 섹션만** 수정 + 변경 이력에 1줄 추가.
- 가이드를 통째로 다시 쓰지 말 것 (버전 추적 손실).
- 가이드 변경 시 QA 보고서에 "가이드 vN 기준으로 테스트" 명시.
